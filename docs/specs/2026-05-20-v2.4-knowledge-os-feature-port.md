# Spec: v2.4.0 — Knowledge OS feature port

**Date:** 2026-05-20
**Status:** Approved direction (Option B)
**Predecessor specs:** `2026-05-20-lint-and-decay-verbs.md` (v2.2), `2026-05-20-adapter-ports-and-followups.md` (v2.3)
**Architecture comparison:** `2026-05-20-dies-vs-dice-architecture.html`

## Decision recap

After surveying the live Knowledge OS ontology (last updated 2026-05-20) and comparing against the AIYO OS DICE spec, **Option B was chosen**: stay on DIES, port four already-thought-through KOS features. Rationale:

- DICE is unproven — not even live in the user's own KOS.
- Team metadata (`owner`, `reviewers`, `scope`, `thread`) is dead weight for Memento OS's solo-developer Path B audience.
- DICE belongs to AIYO OS as a paid-tier commercial moat. Shipping it in OSS Memento OS erodes that wedge.
- The four KOS features deepen the v2.2 enforcement story rather than pivoting schema.

## Goal

Port four features from Knowledge OS ontology into Memento OS, each independently shippable:

| Block | Feature | KOS source |
|-------|---------|------------|
| A | **Funnel Architecture** — `[I]` splits to `_insights.md`; 5K-token area cap | `[D]#431`, 2026-05-20 |
| B | **Project-Class taxonomy** — orthogonal `class:` field, class-level `_context.md` auto-load | ontology line 80–89 (initial classes pre-created 2026-04-26) |
| C | **HEAD/Archive Split** — append-only `_archive.md` companion when cap exceeded | ontology line 135+, "activated 2026-04-19" on AIYO OS |
| D | **Emission Discipline** — only deliberation skills emit `[D]`; research downgrades to `[S]` | ontology line 112–124 |

## Maturity honesty check

Not all four blocks are equally proven in live KOS:

| Block | KOS live status as of 2026-05-20 |
|-------|----------------------------------|
| A — Funnel | Spec only (decided today, zero `_insights.md` files exist yet) |
| B — Project-Class | **Partially live** — 6 class folders + `_context.md` files exist since 2026-04-26 |
| C — HEAD/Archive | Spec only ("activated 2026-04-19 on AIYO OS" but no `_archive.md` exists in vault) |
| D — Emission Discipline | Live as a declarative rule in ontology, not yet enforced by tooling |

Memento OS v2.4 will be the **first concrete implementation** of A, C, and D. This makes Memento OS a useful proving ground for KOS — opposite direction from DICE (where AIYO OS would prove first).

## Constraints (carry over from v2.3.0)

- **No artifact schema rename.** DIES stays. `[D]`/`[I]`/`[E]`/`[S]` grammar is load-bearing.
- **No new package dependencies.** Python 3 stdlib + bash only.
- **No breaking changes to existing vaults.** All new features opt-in. Legacy vaults work as-is.
- **No `memento:format`-type verbs.** Reformatting prose is commodity.
- **Adapter parity required** for any new SKILL.md surface — Codex (Tier 1), Cursor/Windsurf/Cline/Gemini (Tier 2 rules), Aider/Continue (Tier 3 manual).

## Release plan

**Recommended: bundle as v2.4.0.** Each block is small in isolation but together they deepen the enforcement moat in a coherent story ("structured cap shapes, not just structured grammar"). Cleaner CHANGELOG block.

| Version | Scope | Effort | Risk |
|---------|-------|--------|------|
| v2.4.0 | All four blocks A–D | ~6–8 hr | medium — touches session-complete, vault-audit, lint, decay, starter vault, all 8 adapters |

Alternative: ship as patches v2.3.1 (D — declarative only), v2.3.2 (A — file split), v2.3.3 (B — class), v2.3.4 (C — archive). Lower per-release risk; weaker narrative.

## Block A — Funnel Architecture

### What changes

`_context.md` no longer holds `[I]` insights. Insights split to a companion `_insights.md` file co-located with `_context.md`. At area level (`Projects/_context.md`, `Personal/_context.md`, `Business/_context.md`, `Knowledge/_context.md`), the cap shape changes from "24 entries" to **"5K tokens"** — measured as character count divided by 4 as a stable proxy (no `tiktoken` dependency needed).

### File layout after migration

```
Projects/memento-os/
├── _context.md       ← [D]/[S]/[E] only, 24-entry cap (project-level)
├── _insights.md      ← [I] only, no hard cap (target <50 per project)
└── _archive.md       ← (added by Block C) aged-out artifacts
```

### Why this works for Memento OS

The current Memento OS vault cache (`.vault-cache/_context.md`) already has 19 active artifacts after eviction. Two of those (`[I]#16`, `[I]#21`) are insights that lint flagged earlier — they're informational, not decision-bearing. Moving them to `_insights.md` keeps L1 load tight while preserving the learning.

### Lint changes

- Lint walks `_context.md` AND `_insights.md` for any vault. Same rules apply.
- New rule **R8 (warn-only by default)**: `[I]` artifacts in `_context.md` are flagged with "should this be in `_insights.md`?". Promoted to violation under `--strict`.
- `_insights.md` files MUST have frontmatter `type: insights-companion` (new type).

### Decay changes

- Decay also scans `_insights.md` because insights can decay too (the invalidates clause still applies).
- Vault-signal scoring works across both files — a newer `[D]` in `_context.md` can supersede an older `[I]` in `_insights.md`.

### Session-complete changes

- When extracting artifacts, route `[I]` → `_insights.md`. Other types (`[D]`/`[S]`/`[E]`) → `_context.md`.
- Project-level cap stays at 24 entries (for `[D]`/`[S]`/`[E]`). Area-level switches to 5K-token cap (counts chars in Active Reasoning Artifacts section / 4).
- When area cap exceeded → trigger Block C (Archive Sweep), not auto-eviction.

### Migration tooling

New script `skills/memento-funnel-migrate/migrate.py`:
- Walks every `_context.md` in a vault
- Extracts all `[I]` rows
- Creates `_insights.md` adjacent if missing (with frontmatter)
- Moves rows, preserving numbers (or renumbers — TBD per Q1)
- Dry-run by default (`--apply` to commit)
- Idempotent (re-running is a no-op on already-migrated vaults)

### Tests

Fixtures: `tests/fixtures/funnel-pre/` (legacy single-file shape) + `tests/fixtures/funnel-post/` (split shape). Assert:
- `migrate.py --apply` on `funnel-pre/` produces structure matching `funnel-post/`.
- Lint passes on `funnel-post/` with `[I]`-routing rule.
- Lint emits R8 warnings on `funnel-pre/` (when run with `--strict-funnel`).

## Block B — Project-Class Taxonomy

### What changes

New optional frontmatter field on project `_context.md`:

```yaml
class: oss | saas | ios | api-business | vault-only | business | null
```

A class-level `_context.md` lives at `Projects/_types/{class}/_context.md`. When the agent loads context for a project with `class: oss`, it ALSO loads `Projects/_types/oss/_context.md` at L1.5 — between project context and broader vault.

### Starter vault changes

Pre-create 6 class folders under `starter/obsidian-vault/Projects/_types/`:

```
Projects/_types/
├── oss/_context.md            ← seed with template + "rename when first OSS project lands"
├── saas/_context.md
├── ios/_context.md
├── api-business/_context.md
├── vault-only/_context.md
└── business/_context.md
```

Each gets a minimal template (Summary + Active Reasoning Artifacts table) with `project_class: <slug>` frontmatter.

### Session-start changes (memento:session-start)

Add Step 3.5 (L1.5 — Class Context):
- Read project `_context.md` frontmatter
- If `class: X` field present, ALSO read `Projects/_types/X/_context.md` top 3 sections
- Add to briefing: "Class context loaded: X" if applicable

### Session-complete changes

When writing an artifact, ask: "Project-specific, or class-level?" If class-level → route to `Projects/_types/{class}/_context.md` instead of project file.

Routing rule per KOS ontology: an artifact belongs in the class file when *"would this still be true if I started a new project of the same class?"* → YES.

### Lint changes

- Lint walks `Projects/_types/{class}/_context.md` files too. Same rules apply.
- New frontmatter validation R9 (warn): if `class:` is set, the value must be in the registered set.

### Tests

Fixtures: `tests/fixtures/class/Projects/{name1,name2}/_context.md` with `class: oss`, plus `tests/fixtures/class/Projects/_types/oss/_context.md`. Assert lint walks both. Class registry test: invalid class name flagged with R9.

## Block C — HEAD/Archive Split

### What changes

When a `_context.md` exceeds its cap (24 entries project-level OR 5K tokens area-level), `/vault-audit` offers to split:
- Live `_context.md` keeps the working set (Kobe-capped, agent-readable, evictable)
- New `_archive.md` co-located in the same folder, append-only, never auto-loaded

### `_archive.md` schema

```yaml
---
type: reference
status: archived
confidence: high
project: <slug>
created: <date>
---

# Archive — <Project Name>

> Append-only. Era/theme sections, not chronological dump.
> /session-start NEVER loads this file. Grep-on-demand for audits.

## Era: <theme> (<date range>)

| # | Artifact | Priority | Date | Superseded by | Archived |
|---|----------|----------|------|---------------|----------|
| 18 | `[D] README story goes AFTER product/install section` | embedded | 2026-03-21 | embedded in code | 2026-04-14 |
```

### Session-start hard rule

`/session-start` and lint MUST skip files with frontmatter `status: archived`. Implementation: filter `discover_context_files()` in lint.py and decay.py.

### Vault-audit changes

New Step 9 — Archive Sweep:
- Scan each `_context.md` for over-cap state
- Identify settled/superseded entries older than 30 days
- Surface candidates: "Move X to `_archive.md`?"
- On user confirm, move rows + maintain `superseded_by:` backlinks

### Backlink convention

Live `_context.md` entries that supersede archived ones SHOULD carry a `supersedes: [#29, #30, #40]` annotation pointing into `_archive.md`.

### Tests

Fixture: `tests/fixtures/archive/_context.md` (live) + `tests/fixtures/archive/_archive.md`. Assert lint+decay both skip `_archive.md`. Assert vault-audit identifies move candidates correctly.

## Block D — Emission Discipline

### What changes

Every skill declares an `emission_class` in its frontmatter:

```yaml
---
name: memento:decide
emission_class: deliberation   # may emit [D]
---
```

```yaml
---
name: memento:research        # hypothetical future skill
emission_class: research      # may NOT emit [D]; auto-downgrades to [S]
---
```

| Class | Skills | May emit `[D]`? |
|-------|--------|-----------------|
| **deliberation** | `decide`, `grill-me`, `commit-decision`, `spec` | YES |
| **research** | future: `summarize`, `research`, `extract-design`, `audit-website` | NO — downgrade to `[S]` |
| **maintenance** | `vault-audit`, `session-start`, `session-complete`, `lint`, `decay` | pass-through only — may route artifacts produced elsewhere; never originate `[D]` |

### Existing skill updates

Update frontmatter of every existing Memento OS skill:

| Skill | emission_class |
|-------|----------------|
| `memento:decide` | deliberation |
| `memento:grill-me` | deliberation |
| `memento:session-start` | maintenance |
| `memento:session-complete` | maintenance |
| `memento:vault-audit` | maintenance |
| `memento:lint` | maintenance |
| `memento:decay` | maintenance |

### Why this matters

Per the KOS ontology rationale: "_context.md files auto-load into LLM context each session. Speculative [D]s emitted from research cost identical tokens to committed [D]s from deliberation, but the LLM weighs them identically. Gresham's Law of context — low-conviction artifacts crowd out high-conviction ones."

### Lint changes

- Optional new rule **R10 (warn-only)** — if an artifact in `_context.md` carries an `emitted_by: <skill>` metadata field, validate emission class. Requires session-complete to start writing the emitted_by trailer.
- Backward compat: existing artifacts without `emitted_by` are exempt.

### Documentation changes

- `conventions.md` (in starter vault) — new section "Emission Discipline" explaining the rule
- Every adapter SKILL.md / rules file gets an "Emission Discipline" callout box
- README — one paragraph explaining why the rule exists (Gresham's Law)

### Tests

Mostly documentation. One assertion: `lint.py` parses emission_class frontmatter without error. No fixture testing required because the rule is mostly prose.
