DIES vs. DICE — Memento OS vNext architecture

Side-by-side architecture map · 2026-05-20 · DECIDED — Option B (see docs/specs/2026-05-20-v2.4-knowledge-os-feature-port.md)

Two competing artifact schemas. DIES is what Knowledge OS, Memento OS v2.3.0, and every adapter ship today. DICE is a forward-looking schema from the AIYO OS grill session (2026-04-19) — proposed for the paid team-collaboration product, not yet adopted in the live Knowledge OS ontology.

Storage architecture

📁 _context.md — Active Reasoning Artifacts
Single table holds all four types. Kobe cap = 24 entries per project (Memento OS) / 5K-token at area level (Knowledge OS, post-funnel).
↓ over cap
🗄 Decisions/ + eviction
Eviction order: noise → settled → volatile. Never auto-evict [D]. Full-file artifacts in Decisions/ for critical/volatile.

Seed lifecycle (the [S] story)

🌱 Seed planted in _context.md
[S] idea — activates when X sits in the artifact table.
↓ /session-start checks activation
⚡ Activation condition fires
Session-start surfaces in "Seeds Ready". User decides.
↓ /decide --commit
[D] Promoted to decision
Seed becomes a committed [D] with an invalidation trigger.

Cardinal property

  • Single-pane visibility — every reasoning artifact lives in one table, loaded on every session.
  • Solo-optimized — no owner/reviewer/scope metadata.
  • Linear lifecycle — Seed → Decision → (maybe Archived).

DICE — proposed schema

spec — AIYO OS

4 artifact types, S removed, C added

[D]
Decision — same as DIES
priority
date
[I]
Insight — same as DIES
priority
date
[C]
Consequence — paired to [D]#N — manifests when X — mitigation: Y
priority
date
[E]
Error — same as DIES
settled
date

Storage architecture

📁 _context.md — DICE log
Holds [D] + [I] + [C] + [E] only. Rolling token cap, not entry cap. Auto-archive older sessions.
🌱 _ideas/ — seeds live here
All [S]-style "deferred ideas" move OUT of the artifact table into a dedicated _ideas/ folder. Lightweight stubs with depends_on field.
👥 Team metadata on every artifact
New fields: owner, reviewers, scope, status (proposed→accepted→revoked), thread for dissent. Solo users would carry these empty.

Consequence pairing — the new ritual

[D] Decision recorded with invalidator
[D]#3 vendor lock-in — invalidates if pricing doubles
↓ pair the downstream effect
[C] Consequence + mitigation
[C] paired to [D]#3 — manifests when pricing doubles — mitigation: migrate to OSS alternative within Q2
↓ /decay detects trigger fired
⚠️ Mitigation activated
User isn't just told "the decision invalidated" — they see the pre-committed mitigation, ready to execute.

Cardinal property

  • Pre-committed responses — every load-bearing decision carries its mitigation plan.
  • Team-optimized — decisions become contracts between people (owner, reviewers, dissent thread).
  • Branching lifecycle — Idea (in _ideas/) → Decision → either Insight or Consequence.

Recommendation

Don't adopt DICE in Memento OS yet. Port the four already-live Knowledge OS features instead. Revisit DICE when AIYO OS adopts it in production and survives 2–3 months of real team usage.

Why DICE-first is the wrong move for Memento OS

  • DICE is unproven. Not even live in your own Knowledge OS ontology (last updated 2026-05-20, still uses [S]).
  • Team metadata is dead weight for solo OSS users. Memento OS is Path B (solo developer reputation lane). owner/reviewers/scope/thread don't apply.
  • Migration cost falls on existing users. Every vault with [S] artifacts needs tooling to move seeds to _ideas/. Breaking-change tax.
  • It dilutes the v2.2 enforcement story. Memento OS just shipped lint+decay as the differentiator vs. Jeff Su's commodity templates. Adding a schema rename right after fragments the marketing message.
  • DICE serves the wrong product. It's optimized for AIYO OS (paid team product). Putting it in OSS Memento OS gives AIYO OS no commercial moat.

Why the Knowledge OS feature port is the right move

  • Already proven. All four features are live in your active vault. Battle-tested by you for months.
  • Continuous with v2.2. Funnel Architecture and Emission Discipline are more enforcement, not a schema rewrite. Same positioning, sharper edge.
  • No breaking change. Existing vaults work as-is. New features opt-in.
  • Differentiator stacking. v2.2 = "lint+decay" → v2.3 = "+ Funnel + Class + Archive + Emission Discipline." Each release deepens the moat against prose-template competitors.
  • Keeps DICE as a commercial moat for AIYO OS. When AIYO OS launches, "DICE + team collab" is the paid-tier feature gate. If DICE ships in OSS, AIYO OS loses that wedge.

The three paths

Option A — DICE-first

Rename [S] → [C] + move seeds to _ideas/ + add team metadata. Memento OS leads, AIYO OS follows.

When right: You decide Memento OS is the proving ground for AIYO OS. You're willing to break existing user vaults. The "lead the schema" narrative outweighs the commercial-moat-erosion risk.

Option B — Port the live KOS features

Funnel Architecture + Project-Class + HEAD/Archive Split + Emission Discipline. Stay on DIES. Each is independently proven in your daily vault use.

When right: You want v2.4/v3.0 to deepen the enforcement moat, not pivot schema. No breaking change. Keeps DICE as AIYO OS's paid-tier wedge.

Option C — Both

DICE rename AND port the four KOS features in one bundle. Single big release, single big risk.

When right: You've already validated DICE in your own KOS for 4+ weeks. You're ready to ship one large breaking-change release and absorb the support load.
